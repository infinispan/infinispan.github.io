New Functional Map API in Infinispan 8 - Introduction

<p><div dir="ltr" style="text-align: left;" trbidi="on">In Infinispan 8.0.0.Beta3, we have a introduced a new experimental API for interacting with your data which takes advantage of the functional programming additions and improved asynchronous programming capabilities available in Java 8.<br/><br/>Over the next few weeks we'll be introducing different aspects of the API. In this first blog post, we'll focus on why we felt there's a need for a new approach, answering a few key questions.<br/><br/><h2 style="text-align: left;">ConcurrentMap and JCache</h2><br/>Map&#xAD;-like key/value pair APIs have often been used for distributed caching and in-&#xAD;memory data grids. Initially, <a href="https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/ConcurrentMap.html"><span style="font-family: Courier New, Courier, monospace;">ConcurrentMap</span></a> became popular but this was designed to be run within a single JVM, and hence some of the operations suffered in distributed environments or when persistence stores were attached. For example, methods such as '<a href="https://docs.oracle.com/javase/8/docs/api/java/util/Map.html#put-K-V-"><i>V put(K, V)</i></a>', '<a href="https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/ConcurrentMap.html#putIfAbsent-K-V-"><i>V putIfAbsent(K, V)</i></a>', <i>'<a href="https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/ConcurrentMap.html#replace-K-V-">V replace(K, V)</a></i>' would force implementations to return the previous value, but often this value is not needed yet this could be expensive to transfer.<br/><br/><a href="https://jcp.org/en/jsr/detail?id=107">JSR&#xAD;-107</a> set out to improve on this and came up with the <a href="https://github.com/jsr107/jsr107spec/blob/v1.0.0/src/main/java/javax/cache/Cache.java"><span style="font-family: Courier New, Courier, monospace;">JCache</span></a> specification which solved this particular problem separating operations such ConcurrentMap's '<a href="https://docs.oracle.com/javase/8/docs/api/java/util/Map.html#put-K-V-"><i>V put(K, V)</i></a>' into two operations: '<a href="https://github.com/jsr107/jsr107spec/blob/v1.0.0/src/main/java/javax/cache/Cache.java#L194"><i>void put(K, V)</i></a>' and '<i><a href="https://github.com/jsr107/jsr107spec/blob/v1.0.0/src/main/java/javax/cache/Cache.java#L224">V getAndPut(K, V)</a>'</i>, and it applied the same logic to other operations such as '<i>replace</i>' by providing an alternative '<i><a href="https://github.com/jsr107/jsr107spec/blob/v1.0.0/src/main/java/javax/cache/Cache.java#L459">getAndReplace(K, V)</a>'</i>… etc.<br/><br/>However, even though JCache was designed with distributed caching in mind, it still failed to provide an API to execute operations asynchronously and hence avoid resource under&#xAD;utilization by having threads waiting for remote operations to complete. '<i><a href="https://github.com/jsr107/jsr107spec/blob/v1.0.0/src/main/java/javax/cache/Cache.java#L169">l&#x200B;oadAll</a>'</i>&#x200B;is probably the only exception, and it would have been the perfect candidate to return a <a href="http://docs.oracle.com/javase/8/docs/api/java/util/concurrent/Future.html">F&#x200B;uture&#x200B;</a> or similar construct, but having to pass in a completion listener feels a bit clunky and cannot be chained easily.<br/><br/>In my opinion, the best parts of JCache are '<i><a href="https://github.com/jsr107/jsr107spec/blob/v1.0.0/src/main/java/javax/cache/Cache.java#L559">i&#x200B;nvoke</a>'</i>&#x200B; and '<a href="https://github.com/jsr107/jsr107spec/blob/v1.0.0/src/main/java/javax/cache/Cache.java#L599"><i>i&#x200B;nvokeAll</i></a>' methods. When you<br/>look at them, you see a lot of potential to reimplement <i>get</i>, <i>put</i>, <i>getAndPut</i>, <i>getAndReplace</i>, <i>putAll</i>,&#x200B; <i>getAll</i>, &#x200B;and many others using these methods. In other words, as an implementer, all you should need to implement is those two functions, and the rest would be syntactic sugar for the user. Unfortunately, the way '<i>i&#x200B;nvoke</i>' and '<i>i&#x200B;nvokeAll</i>' handle arguments is a bit clunky, and really, it's just screaming for lambdas to be passed in and <a href="https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/CompletableFuture.html"><span style="font-family: Courier New, Courier, monospace;">C&#x200B;ompletableFuture</span></a>instancesto be returned (Java 8!).<br/><br/>So, when Infinispan moved to Java 8, we decided to revisit these concepts and see if we could come up with a better, distilled map&#xAD;-like interface to be used for as either a caching or data grid API.<br/><br/><h2 style="text-align: left;">New Functional Map API</h2><br/>Infinispan's <a href="https://github.com/infinispan/infinispan/blob/master/commons/src/main/java/org/infinispan/commons/api/functional/FunctionalMap.java">Functional Map API</a> is a distilled map&#xAD;like asynchronous API which uses lambdas to interact with data.<br/><h4 style="text-align: left;">Asynchronous and Lazy</h4>Being an asynchronous API, all methods that return a single result, return a <a href="https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/CompletableFuture.html"><span style="font-family: Courier New, Courier, monospace;">CompletableFuture</span></a> which wraps the result, so you can use the resources of your system more efficiently by having the possibility to receive callbacks when the <span style="font-family: Courier New, Courier, monospace;">CompletableFuture</span> has completed, or you can chain or compose them with other <span style="font-family: Courier New, Courier, monospace;">CompletableFuture</span>. If you do want to block the thread and wait for the result, just as it happens with a <span style="font-family: Courier New, Courier, monospace;">ConcurrentMap</span> or <span style="font-family: Courier New, Courier, monospace;">JCache</span> method call, you can simply call &lt;a href="https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/CompletableFuture.html#get--"&gt;&lt;i&gt;CompletableFuture.get()&lt;/i&gt;&lt;/a&gt; (for such situations, we are working on finding ways to avoid unnecessary thread creation when the caller will block on the <span style="font-family: Courier New, Courier, monospace;">CompletableFuture</span>).<br/><br/>For those operations that return multiple results, the API returns instances of a &#x200B;<a href="https://github.com/infinispan/infinispan/blob/master/commons/src/main/java/org/infinispan/commons/api/functional/Traversable.java"><span style="font-family: Courier New, Courier, monospace;">Traversable</span></a> interface which offers a lazy pull&#xAD;-style API for working with multiple results. Although push&#xAD;-style interfaces for handling multiple results, such as RxJava, are fully asynchronous, they're harder to use from a user&#x2019;s perspective. <span style="font-family: Courier New, Courier, monospace;">T&#x200B;raversable</span>,&#x200B; being a lazy pull&#xAD;-style API, can still be asynchronous underneath since the user can decide to work on the traversable at a later stage, and the <span style="font-family: Courier New, Courier, monospace;">Traversable</span> implementation itself can decide when to compute those results.<br/><h4 style="text-align: left;">Lambda transparency</h4>Since the content of the lambdas is transparent to Infinispan, the API has been split into 3 interfaces for read&#xAD;-only (<a href="https://github.com/infinispan/infinispan/blob/master/commons/src/main/java/org/infinispan/commons/api/functional/FunctionalMap.java#L106"><span style="font-family: Courier New, Courier, monospace;">R&#x200B;eadOnlyMap</span></a>)&#x200B;, read&#xAD;-write (<a href="https://github.com/infinispan/infinispan/blob/master/commons/src/main/java/org/infinispan/commons/api/functional/FunctionalMap.java#L456"><span style="font-family: Courier New, Courier, monospace;">R&#x200B;eadWriteMap</span></a>)&#x200B; and write&#xAD;-only (<a href="https://github.com/infinispan/infinispan/blob/master/commons/src/main/java/org/infinispan/commons/api/functional/FunctionalMap.java#L227"><span style="font-family: Courier New, Courier, monospace;">W&#x200B;riteOnlyMap</span></a>)&#x200B; operations respectively, in order to provide hints to the Infinispan internals on the type of work needed to support lambdas.<br/><br/>For example, Infinispan has been designed in such way that our '<i>C&#x200B;oncurrentMap.&#x200B;g&#x200B;et(&#x200B;)</i>' and '<i>JCache.&#x200B;g&#x200B;etAll(&#x200B;)</i>' implementations do not require locks to be acquired. These <i>get()</i>/<i>getAll()</i>operations are read-only operations, and hence if you call our functional map <span style="font-family: Courier New, Courier, monospace;">R&#x200B;eadOnlyMap</span><span style="font-family: inherit;">'s</span>'<i><a href="https://github.com/infinispan/infinispan/blob/master/commons/src/main/java/org/infinispan/commons/api/functional/FunctionalMap.java#L144">eval(&#x200B;)</a></i>' or '<i><a href="https://github.com/infinispan/infinispan/blob/master/commons/src/main/java/org/infinispan/commons/api/functional/FunctionalMap.java#L176">e&#x200B;valMany(&#x200B;)</a></i>' operations, you get the same benefit. A key advantage of <span style="font-family: Courier New, Courier, monospace;">R&#x200B;eadOnlyMap</span><span style="font-family: inherit;">'s</span> '<i>eval&#x200B;()</i>' and '<i>e&#x200B;valMany(&#x200B;)</i>' operations is that they take lambdas as parameters which means the returned types are more flexible, so we can return a value associated with the key, or we can return a boolean if a value has the expected contents, or we can return some metadata parameters from it, e.g. last accessed time, last modified time, creation time, lifespan, version information…etc.<br/><br/>Another important hint that is required to make efficient use of the system is to know when a write-only operation is being executed. Write&#xAD;-only operations require locks to be acquired and as demonstrated by <span style="font-family: Courier New, Courier, monospace;">JCache</span>'s '<a href="https://github.com/jsr107/jsr107spec/blob/v1.0.0/src/main/java/javax/cache/Cache.java#L505"><i>void removeAll()</i></a>' and `<i><a href="https://github.com/jsr107/jsr107spec/blob/v1.0.0/src/main/java/javax/cache/Cache.java#L194">void put(K, V)</a></i>' or <span style="font-family: Courier New, Courier, monospace;">ConcurrentMap</span>'s '<a href="https://docs.oracle.com/javase/8/docs/api/java/util/Map.html#putAll-java.util.Map-"><i>putAll()</i></a>', they do not require the previous value to be queried or read, which as explained above is a very important optimization since reading the previous value might require the persistence layer or a remote node to be queried. <span style="font-family: Courier New, Courier, monospace;">WriteOnlyMap</span>'s '<a href="https://github.com/infinispan/infinispan/blob/master/commons/src/main/java/org/infinispan/commons/api/functional/FunctionalMap.java#L281"><i>eval()</i></a>', '<a href="https://github.com/infinispan/infinispan/blob/master/commons/src/main/java/org/infinispan/commons/api/functional/FunctionalMap.java#L351"><i>evalMany()</i></a>', and '<a href="https://github.com/infinispan/infinispan/blob/master/commons/src/main/java/org/infinispan/commons/api/functional/FunctionalMap.java#L414"><i>evalAll()</i></a>' follow this same pattern with the added flexibility for the lambda to decide what kind of write operation to execute.<br/><br/>The final type of operations we have are read&#xAD;-write operations, and within this category we find CAS-like (Compare&#xAD;-And&#xAD;-Swap) operations. This type of operations require previous value associated with the key to be read and for locks to be acquired before executing the lambda. &#xFFFC;Most of the operations in <span style="font-family: Courier New, Courier, monospace;">ConcurrentMap</span> and <span style="font-family: Courier New, Courier, monospace;">JCache</span> operations fall within this domain including: '<i><a href="https://docs.oracle.com/javase/8/docs/api/java/util/Map.html#put-K-V-">V put(K, V)</a></i>', '<a href="https://github.com/jsr107/jsr107spec/blob/v1.0.0/src/main/java/javax/cache/Cache.java#L283"><i>boolean putIfAbsent(K, V)</i></a>', '<i><a href="https://docs.oracle.com/javase/8/docs/api/java/util/Map.html#replace-K-V-">V replace(K, V)</a></i>', '<i><a href="https://docs.oracle.com/javase/8/docs/api/java/util/Map.html#replace-K-V-V-">boolean replace(K, V, V)</a></i>'…etc. <span style="font-family: Courier New, Courier, monospace;">ReadWriteMap</span>'s '<i><a href="https://github.com/infinispan/infinispan/blob/master/commons/src/main/java/org/infinispan/commons/api/functional/FunctionalMap.java#L554">eval()</a></i>', '<i><a href="https://github.com/infinispan/infinispan/blob/master/commons/src/main/java/org/infinispan/commons/api/functional/FunctionalMap.java#L581">evalMany()</a></i>' and '<i><a href="https://github.com/infinispan/infinispan/blob/master/commons/src/main/java/org/infinispan/commons/api/functional/FunctionalMap.java#L626">evalAll()</a></i>' provide a way to implement the vast majority of these operations thanks to the flexibility of the lambdas passed in. So you can make CAS&#xAD;-like comparisons not only based on value equality but based on metadata parameter equality such as version information, and you can send back previous value or boolean instances to signal whether the CAS&#xAD;-like comparison succeeded.<br/><br/><h2 style="text-align: left;">$DEITY, I need to learn a new API!!!</h2><br/>This new functional Map&#xAD;-like API is meant to complement existing Key/Value Infinispan API offerings, so you'll still be able to use <span style="font-family: Courier New, Courier, monospace;">ConcurrentMap</span> or <span style="font-family: Courier New, Courier, monospace;">JCache</span> standard APIs if that's what suits your use case best.<br/><br/>The target audience for this new API is either:<br/><ol style="text-align: left;"><li>Distributed or persistent caching/in&#xAD;-memory&#xAD; data&#xAD;-grid users that want to benefit from <span style="font-family: Courier New, Courier, monospace;">CompletableFuture</span> and/or <span style="font-family: Courier New, Courier, monospace;">Traversable</span> for async/lazy data grid or caching data manipulation. The clear advantage here is that threads do not need to be idle waiting for remote operations to complete, but instead these can be notified when remote operations complete and then chain them with other subsequent operations.</li><li>Users wanting to go beyond the standard operations exposed by <span style="font-family: Courier New, Courier, monospace;">ConcurrentMap</span> and <span style="font-family: Courier New, Courier, monospace;">JCache</span>, for example, if you want to do a replace operation using metadata parameter equality instead of value equality, or if you want to retrieve metadata information from values…etc.</li></ol>Internally, we feel that this new functional Map&#xAD;-like API distills the Map&#xAD;-like APIs that we currently offer (including <span style="font-family: Courier New, Courier, monospace;">ConcurrentMap</span> and <span style="font-family: Courier New, Courier, monospace;">JCache</span>) and gets rid of a lot of duplication in our <a href="https://docs.jboss.org/infinispan/8.0/apidocs/org/infinispan/AdvancedCache.html">AdvancedCache</a> API (e.g. '<a href="https://docs.jboss.org/infinispan/8.0/apidocs/org/infinispan/AdvancedCache.html#getCacheEntry-java.lang.Object-"><i>getCacheEntry()</i></a>', '<a href="https://docs.jboss.org/infinispan/8.0/apidocs/org/infinispan/commons/api/AsyncCache.html#getAsync-K-"><i>getAsync()</i></a>', '<a href="https://docs.jboss.org/infinispan/8.0/apidocs/org/infinispan/commons/api/AsyncCache.html#putAsync-K-V-"><i>putAsync()</i></a>', '<i><a href="https://docs.jboss.org/infinispan/8.0/apidocs/org/infinispan/AdvancedCache.html#put-K-V-org.infinispan.metadata.Metadata-">put(K, V, Metadata)</a></i>'…etc), and hence down the line, we'd want all these APIs to be implemented using the new functional Map&#xAD;like API. By doing that, we hope to reduce the number of commands that our internal architecture implements, hence reducing our code base.<br/><br/>This new API also offers a new approach for passing per-invocation parameters, and much more flexible Metadata handling compared to our current approach. As we dig into this new API in next blog posts, we'll explain the differences and advantages provided by these.<br/><br/><h2 style="text-align: left;">Functional Map API usage examples</h2><br/>To
